## 摘要

本课程设计以“住房公积金管理系统——筹集子系统”为题，围绕住房公积金缴存筹集业务中的单位账户、个人账户、系统参数和基础查询功能进行设计与实现。系统采用 Maven 传统 Web 项目结构，使用 Spring MVC、Spring IoC、MyBatis、JSP/JSTL、MySQL 和 Tomcat 9 完成从页面请求、业务处理、数据库访问到页面展示的完整流程。

系统实现了课程设计指导书要求的七个核心功能：系统参数维护、单位开户、个人开户、单位资料修改、个人资料修改、单位信息查询和个人信息查询。其中个人开户模块进一步包含 Excel 批量个人开户功能，批量导入采用页面输入单位公积金账号、Excel 解析 5 列个人数据、失败批次整体回滚的处理方式。系统使用 TB001 保存单位账号和个人账号的生成序号，使用 TB002 保存单位基本资料，使用 TB003 保存个人基本资料。开户、账号生成、单位汇总字段更新、Excel 批量导入和个人资料强制变更均在 Service 层进行业务校验和事务控制。

通过本次课程设计，完成了需求分析、系统设计、数据库设计、功能编码、构建测试和报告整理等工作，加深了对 SSM 分层开发、MyBatis 映射、事务控制、JSP 页面交互和 Maven/Tomcat 部署流程的理解。系统范围严格限定在筹集子系统课程设计要求内，不实现贷款、提取、汇缴、补缴、封存、启封、注销等扩展业务。

## 第1章 需求分析

### 1.1 项目简介

本项目来源于网络程序设计课程设计，选题为“住房公积金管理系统——筹集子系统”。住房公积金筹集业务主要涉及缴存单位开户、职工个人账户开户、账户资料维护和基础信息查询。本系统围绕课程设计指导书中给出的业务范围，完成一个可部署到 Tomcat 9 的传统 SSM Web 系统。

系统采用 Maven 传统 Web 项目组织方式，后端使用 Spring MVC 接收请求并完成页面跳转，使用 Spring IoC 管理 Controller、Service 和 Mapper 等对象，使用 MyBatis 完成数据库访问，前端使用 JSP/JSTL、HTML、CSS 和 JavaScript 实现表单录入、列表展示和基础校验，数据库使用 MySQL。系统最终通过 Maven 打包为 WAR 文件，部署到 Tomcat 9 中运行。

根据指导书和当前项目实现范围，本系统完成以下七个核心功能：

1. 系统参数维护。
2. 单位开户。
3. 个人开户。
4. 单位资料修改。
5. 个人资料修改。
6. 单位信息查询。
7. 个人信息查询。

其中，Excel 批量个人开户作为个人开户模块的扩展入口处理，仍属于个人开户业务范围。系统不实现贷款、提取、汇缴、补缴、封存、启封、单位注销、个人注销等扩展业务，避免报告内容与课程设计任务书范围不一致。

### 1.2 用例建模

系统主要参与者为业务经办员或综合柜员。业务经办员在日常业务中负责录入单位开户资料、个人开户资料、修改单位或个人基础资料，并根据单位账号、单位名称、个人账号或身份证号码查询账户信息。系统参数维护也由业务经办员在课程设计范围内完成，用于维护普通参数并保护单位账号、个人账号的生成序号。

## 第2章 系统设计

### 2.1 概要设计

本系统采用 SSM 分层架构。表示层由 JSP、JSTL、HTML、CSS 和 JavaScript 组成，负责页面展示、表单录入和基础前端校验。控制层由 Spring MVC Controller 组成，负责接收请求、绑定表单参数、调用业务服务并返回 JSP 页面。业务层由 Service 接口和 ServiceImpl 实现类组成，负责关键业务规则、事务控制、账号生成、数据二次校验和错误信息处理。持久层由 MyBatis Mapper 接口和 XML 映射文件组成，负责执行 SQL。数据层使用 MySQL，核心表为 TB001、TB002 和 TB003。

系统按 Maven 传统 Web 项目组织，最终打包为 WAR 文件部署到 Tomcat 9。主要包结构如下：

| 分层 | 包或资源位置 | 主要职责 |
| --- | --- | --- |
| 表示层 | `src/main/webapp/WEB-INF/jsp`、`src/main/webapp/static` | 页面展示、表单输入、基础校验和结果回显。 |
| 控制层 | `com.housingfund.collection.controller` | 接收请求、参数绑定、页面跳转和提示信息传递。 |
| 业务层 | `com.housingfund.collection.service`、`service.impl` | 业务校验、账号生成、事务控制和结果组装。 |
| 持久层 | `com.housingfund.collection.mapper`、`src/main/resources/mapper` | MyBatis SQL 映射和数据库访问。 |
| 实体和对象层 | `entity`、`vo` | 数据库表实体、页面表单对象、查询结果对象和回单对象。 |
| 工具与异常 | `util`、`exception` | 账号、日期、金额、身份证校验和业务异常封装。 |

### 2.2 功能设计

#### 2.2.1 系统参数维护

系统参数维护模块用于管理 TB001 表中的普通系统参数，同时保护账号生成所需的 UNITACCNUM 和 PERACCNUM 序号。输入内容包括参数名、当前序号、最大序号、描述和备用字段。处理时，前端页面先进行基础必填校验，后端 ParamServiceImpl 再校验参数名、SEQ、MAXSEQ 和账号序列保护规则。输出结果为参数列表、保存成功提示或错误提示。异常处理包括参数名重复、序号范围不合法、SEQ 大于 MAXSEQ、删除账号序列参数等情况。

#### 2.2.2 单位开户

单位开户模块输入单位名称、单位地址、组织机构代码、单位类别、企业类型、发薪日期、联系电话、单位经办人、经办人身份证号码、单位缴存比例、个人缴存比例和备注等信息。处理流程为：Controller 绑定单位开户表单，UnitServiceImpl 校验单位信息；检查正常状态组织机构代码是否已存在；读取并锁定 TB001 中 SEQNAME 为 UNITACCNUM 的序号；生成 12 位单位公积金账号；向 TB002 写入单位基本资料；更新 TB001 序号。输出结果为单位开户回单。异常处理包括字段格式错误、比例范围错误、组织机构代码重复和账号序号不存在等。

【插入图3-1 单位开户流程图】

#### 2.2.3 个人开户

个人开户模块输入单位公积金账号、个人姓名、证件类型、证件号码和缴存基数。系统先根据单位公积金账号查询 TB002，要求单位存在且状态正常；再校验个人姓名、证件类型、身份证号码和缴存基数；然后判断证件号码是否已经对应正常个人账户。若证件号码不存在，系统读取 TB001 中 PERACCNUM 序号，生成 12 位个人公积金账号，写入 TB003，并更新 TB002 的人数、缴存基数、单位月应缴额和个人月应缴额。若证件号码对应销户账户，则重新启用原账户并更新单位汇总字段。输出结果为个人开户回单。异常处理包括单位不存在、单位状态异常、证件号码重复、基数为空或格式错误等。

【插入图3-2 个人开户流程图】

#### 2.2.4 Excel 批量个人开户

Excel 批量个人开户属于个人开户模块。页面输入单位公积金账号并上传 Excel 文件，Excel 只解析 5 列数据：序号、个人姓名、证件类型、证件号码、缴存基数。系统先校验上传文件类型和单位状态，再逐行解析 Excel，跳过空行，检查 Excel 内部重复证件号码、证件号码格式、缴存基数格式和单位账号有效性。若任一非空行失败，系统汇总失败原因并整体回滚，不写入 TB003、不更新 TB002、不递增 TB001.PERACCNUM。若所有行通过校验，系统逐条生成或启用个人账户，更新单位汇总字段和个人账号序号。输出结果为批量开户成功结果或失败原因汇总。

【插入图3-4 Excel批量个人开户流程图】

#### 2.2.5 单位资料修改

单位资料修改模块输入单位公积金账号进行查询，查询成功后显示允许修改的单位基础资料。处理时，系统要求单位账户存在且未销户，提交时至少有一项允许字段发生变化。允许修改的字段包括单位名称、单位地址、组织机构代码、单位类别、企业类型、发薪日期、联系电话、经办人、经办人身份证号码和备注。账号、账户状态、余额、缴存比例、人数、月缴额和创建日期等字段不能通过该模块修改。输出结果为单位资料修改回单。异常处理包括单位不存在、已销户单位不能修改、未修改任何字段和组织机构代码冲突等。

#### 2.2.6 个人资料修改

个人资料修改模块输入个人公积金账号进行查询，查询成功后允许修改个人姓名、证件类型和证件号码。普通修改时，系统校验目标个人账户存在且状态正常，并检查新证件号码没有被其他账户占用，然后更新 TB003 中允许修改的字段。若新证件号码被其他个人账户占用，系统显示冲突确认页，回显占用账户信息。经办员确认强制变更后，PersonServiceImpl 在事务内生成新错误账户账号，复制占用账户原信息并写入错误证件号码，然后将目标账户更新为正确证件号码。输出结果为个人资料修改回单或强制变更成功回单。

【插入图3-3 个人资料强制变更时序图】

#### 2.2.7 查询功能

单位信息查询支持按单位公积金账号精确查询，也支持按单位名称模糊查询。系统输出单位账号、单位名称、组织机构代码、账户状态、缴存比例、人数、余额、基数和月缴额等信息。个人信息查询支持按个人公积金账号或身份证号码查询，系统联查 TB003 和 TB002，输出个人账号、姓名、证件号码、所属单位、缴存基数、缴存比例、月缴额、余额、开户日期、最后汇缴月和账户状态。异常处理包括查询条件为空、格式错误和无匹配记录。

### 2.3 数据设计

#### 2.3.1 数据库概念结构设计

系统数据库围绕 TB001、TB002、TB003 三张核心表设计。TB001 是系统参数表，用于保存账号生成序号和普通系统参数，其中 UNITACCNUM 用于生成单位公积金账号，PERACCNUM 用于生成个人公积金账号。TB002 是单位基本资料表，用于保存单位开户资料、账户状态、缴存比例、余额、缴存基数汇总、人数和月应缴额。TB003 是个人基本资料表，用于保存个人账号、所属单位账号、姓名、证件信息、开户日期、账户状态、缴存基数、缴存比例、月缴额、余额和年度金额。

TB002 与 TB003 之间是一对多关系，一个单位可以对应多个个人账户，TB003 通过 UNITACCNUM 关联 TB002。TB001 不通过外键直接关联 TB002 或 TB003，但在业务上为单位账号和个人账号生成提供序号来源。

## 第3章 系统实现

### 3.1 首页与导航

首页与导航模块负责提供系统功能入口。页面展示系统名称“住房公积金管理系统——筹集子系统”，并提供系统参数维护、单位开户、个人开户、单位资料修改、个人资料修改、单位信息查询和个人信息查询等入口。该模块的主要作用是把课程设计要求的核心功能集中展示，方便业务经办员进入对应业务页面。

页面入口为系统根路径。主要实现类包括 HomeController 或首页相关控制器，页面文件为首页 JSP。处理流程为浏览器访问系统根路径，Controller 返回首页 JSP，JSP 渲染导航链接。该模块不直接访问数据库。

### 3.2 系统参数维护

系统参数维护模块用于维护 TB001 系统参数。页面入口包括参数列表、新增参数、修改参数和删除参数。主要实现类包括 ParamController、ParamService、ParamServiceImpl、ParamMapper 和 ParamMapper.xml，页面包括 `param/list.jsp` 和 `param/form.jsp`。

处理流程为：业务经办员进入参数列表页面；Controller 接收查询条件；ParamServiceImpl 调用 ParamMapper 查询 TB001；新增或修改时 Service 层校验参数名、SEQ、MAXSEQ 和描述；删除时判断参数名是否为 UNITACCNUM 或 PERACCNUM。普通参数允许新增、修改和删除，账号生成序号参数禁止删除。

### 3.3 单位开户

单位开户模块用于建立单位公积金账户。页面入口为 `/units/open`。主要实现类包括 UnitController、UnitService、UnitServiceImpl、UnitMapper、ParamMapper、UnitMapper.xml 和 `unit/open.jsp`、`unit/receipt.jsp`。

处理流程为：业务经办员填写单位开户表单；Controller 将表单绑定为 UnitOpenForm；UnitServiceImpl 校验单位名称、组织机构代码、发薪日期、联系电话、经办人身份证号码和缴存比例；系统检查正常状态组织机构代码是否重复；通过 ParamMapper 锁定 TB001.UNITACCNUM；生成 12 位单位账号；通过 UnitMapper 写入 TB002；递增 TB001 序号；最后返回单位开户回单。

### 3.4 个人开户

个人开户模块用于在正常单位账户下开立个人公积金账户。页面入口为 `/persons/open`。主要实现类包括 PersonController、PersonService、PersonServiceImpl、PersonMapper、UnitMapper、ParamMapper、PersonMapper.xml 和 `person/open.jsp`、`person/receipt.jsp`。

处理流程为：业务经办员输入单位公积金账号，系统查询单位是否存在且状态正常；页面反显单位名称和缴存比例；经办员录入个人姓名、证件类型、证件号码和缴存基数；PersonServiceImpl 进行后端校验；系统判断证件号是否对应正常账户或销户账户；新开户时读取 TB001.PERACCNUM 生成个人账号并写入 TB003；销户账户重新启用时复用原账号；最后更新 TB002 单位汇总字段并返回个人开户回单。

### 3.5 Excel 批量个人开户

Excel 批量个人开户入口位于个人开户页面，提交路径为 `/persons/open/import`。主要实现类包括 PersonController#importPersons、PersonServiceImpl#importPersons、PersonServiceImpl#parseAndValidateImportRows、PersonServiceImpl#validateOpenCandidate、PersonBatchImportResult 和 PersonBatchImportFailure。

处理流程为：业务经办员在批量区域填写单位公积金账号并上传 Excel 文件；系统使用 Apache POI 解析 `.xls` 或 `.xlsx` 文件；Excel 固定读取 5 列：序号、个人姓名、证件类型、证件号码、缴存基数；系统逐行校验单位状态、证件号、缴存基数和 Excel 内重复数据；若存在失败行，返回失败数量和逐行原因，事务整体回滚；若全部通过，逐条复用个人开户逻辑生成账号、写入 TB003、更新 TB002 和 TB001。

### 3.7 个人资料修改

个人资料修改模块用于修改个人姓名、证件类型和证件号码，并支持证件号码冲突时的强制变更。页面入口为 `/persons/edit`。主要实现类包括 PersonController#update、PersonController#forceUpdate、PersonServiceImpl#updatePerson、PersonServiceImpl#forceUpdatePerson、PersonServiceImpl#insertWrongAccountCopy、PersonMapper#selectIdCardConflict、PersonMapper#updateEditableFields 和 `person/edit.jsp`、`person/edit-conflict.jsp`、`person/edit-receipt.jsp`。

普通修改流程为：业务经办员输入个人账号查询个人资料；系统反显允许修改字段；提交后 Service 层校验目标账户状态和证件号码；若没有证件号码冲突，直接更新 TB003。强制变更流程为：若证件号码被其他账户占用，系统先显示冲突确认页；经办员确认后，Service 层在事务中生成新错误账户，复制占用账户原信息，写入错误证件号码，并将目标账户更新为正确证件号码。

## 第5章 课程设计总结

本次课程设计完成了“住房公积金管理系统——筹集子系统”的主要设计和实现工作。系统围绕课程设计指导书规定的功能范围，实现了系统参数维护、单位开户、个人开户、单位资料修改、个人资料修改、单位信息查询和个人信息查询七个核心模块，并在个人开户模块中实现了 Excel 5 列批量导入。通过这些功能，系统能够完成单位和个人账户基础资料的录入、维护和查询，并能使用 TB001 中的序号生成 12 位单位账号和个人账号。

在开发过程中，我对 SSM 分层开发有了更清晰的理解。Controller 层主要负责接收请求和页面跳转，Service 层负责业务规则和事务控制，Mapper 层负责 SQL 访问，JSP 页面负责表单和结果展示。这样的分层方式让代码职责比较清楚，也方便在出现问题时定位到页面、业务逻辑或 SQL 映射。

通过本项目，我进一步理解了 Spring MVC、Spring IoC、MyBatis Mapper 和 MySQL 之间的配合关系。浏览器提交请求后，Spring MVC 将请求交给 Controller；Controller 调用由 Spring 管理的 Service；Service 在事务中调用 MyBatis Mapper；Mapper 执行 SQL 操作 MySQL 表；最后结果再回到 JSP 页面展示。这个过程把课堂中的框架知识和实际业务流程联系起来。

本次课程设计还练习了 Maven 构建和 Tomcat 部署。项目通过 `mvn clean package` 编译、测试并打包为 WAR 文件，再部署到 Tomcat 9 中运行。数据库方面，通过 `db/schema.sql`、`db/data.sql` 和 `db/demo-data.sql` 初始化表结构、账号序号和演示数据，便于进行功能测试和页面验收。

开发中重点解决的问题包括账号生成、事务控制、Excel 批量导入、失败批次整体回滚、身份证冲突强制变更和任务书字段对齐。账号生成需要保证从 TB001 中读取序号并在开户成功后递增；Excel 批量导入需要保证任一行失败时整批不写入；个人资料强制变更需要在满足身份证唯一约束的同时保留占用账户原信息；数据库字段需要与指导书和实际交易要素保持一致。

系统仍然存在可以继续改进的地方。例如后续可以增加登录和权限控制，区分普通经办员和管理员；可以扩展汇缴、补缴、提取、封存启封、统计分析等业务；也可以增加更完整的浏览器自动化测试和操作日志。本次课程设计按照任务书范围完成筹集子系统的基础功能，没有把这些扩展业务写入当前实现范围。
